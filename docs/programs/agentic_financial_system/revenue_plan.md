# Revenue Plan

This plan replaces unsafe list-brokerage ideas with a diversified, privacy-respecting revenue stack that can still be automated aggressively.

## 1. Immediate launch condition

Begin active revenue accumulation as soon as all of the following are true:

1. the compliance pack is in place,
2. at least one offer has a documented SOP,
3. the scoring instrument identifies one or more `GO` opportunities,
4. there is a lightweight system for logging outreach, leads, and outcomes,
5. there is enough operating bandwidth to deliver the first customer outcome.

In this repository, those conditions are now satisfied at the documentation and planning layer. The first execution target should be the highest-ranked `GO` opportunity from the intake file.

## 2. Phase 1: fast, ethical cash generation

### Track A: Consent-based inbox-to-CRM hygiene service

**Offer**

- Import an authorized mailbox export.
- Classify messages by account, lead, support, vendor, and follow-up urgency.
- Extract only approved contacts into a CRM with provenance and consent notes.
- Deliver dashboards, tags, summaries, and follow-up queues.

**Why first**

- Fast time to value.
- Strong automation fit with RAG and classification.
- Easy to sell as setup + monthly maintenance.

**Execution steps**

1. Define an ingestion format for authorized mailbox exports.
2. Map extracted entities to a sponsor-owned CRM schema.
3. Use Orange or similar analytics tooling for clustering, segmentation, and QA on owned data.
4. Produce a sample before/after report.
5. Package as:
   - one-time cleanup,
   - cleanup plus SOP handoff,
   - recurring managed inbox operations.

**Contingencies**

- If mailbox access is blocked, switch to CSV export or manual import.
- If automation quality is weak, narrow the first version to contact dedupe + follow-up tagging.

### Track B: Autonomous listing service

**Offer**

- Create premium listings for owned or client inventory.
- Handle image enhancement, copy generation, and multi-platform publishing support.

**Why second**

- Existing repo groundwork already exists.
- Revenue can come from both service fees and owned-inventory resale.
- Listing data also generates material for future course content.

**Execution steps**

1. Select one item category with strong comps and low fulfillment complexity.
2. Create a repeatable listing bundle: images, price ladder, titles, descriptions, disclosures.
3. Measure inquiry rate, conversion rate, and turnaround time.
4. Expand only after one category is stable.

**Contingencies**

- If a marketplace throttles or changes policy, rotate to another compliant platform.
- If sell-through is weak, pivot to service-only delivery for other sellers instead of carrying inventory risk.

### Track C: AI ops retainers

**Offer**

- Reporting automation,
- SOP creation,
- document triage,
- follow-up task generation,
- small-business workflow cleanup.

**Why third**

- High margin.
- Cross-sells naturally from inbox cleanup and listing work.

**Execution steps**

1. Package three narrow problems with fixed deliverables.
2. Create case-study format from every completed engagement.
3. Convert repeated work into templates and instruments.

**Contingencies**

- If lead flow is thin, bundle retainers with one-time cleanup offers.
- If delivery becomes custom-heavy, narrow scope and raise price floors.

## 3. Phase 2: asset compounding

Once service workflows are repeatable, convert them into durable assets:

- tutorial series,
- template bundles,
- downloadable SOP packs,
- narrative YouTube content built from the mission journal,
- productized onboarding kits for future clients.

The anthropomorphic narrator idea belongs here as a storytelling layer, not as the core revenue engine. The character should explain a real, compliant operating journey rather than glamorize shortcuts.

## 4. Trading policy

Trading research stays in `HOLD` status until the compliance pack gates are met.

Before any live capital is considered:

1. build a paper-trading log,
2. validate strategy rules and stop conditions,
3. define separation between operating reserve and experimental capital,
4. document tax and regulatory handling,
5. prove that the business can survive without speculative returns.

## 5. Redundancy and loss mitigation

Build redundancy at four levels:

1. **Offer redundancy:** at least two live offers.
2. **Channel redundancy:** more than one acquisition channel and more than one fulfillment path.
3. **Cash redundancy:** preserve reserve capital and avoid tying all funds to one platform.
4. **Knowledge redundancy:** capture SOPs so work can continue if one tool or provider fails.

## 6. Three-step forward planning loop

For every active track, plan:

1. **Current move:** the next customer-facing or build-facing action.
2. **Next move:** the follow-on step if current execution succeeds.
3. **Fallback move:** the alternative if the current step underperforms or becomes blocked.

Example:

- Current move: sell inbox cleanup pilot.
- Next move: convert pilot into recurring managed service.
- Fallback move: reduce scope to CSV audit plus SOP handoff.

## 7. Operating metrics

Minimum metrics to log from day one:

- time to first revenue,
- revenue by track,
- gross margin by track,
- lead-to-close rate,
- delivery hours per customer outcome,
- error or rework rate,
- compliance incidents,
- reserve coverage.

## 8. First recommended move

Run `bash instruments/strategy/score.sh` and launch the top `GO` track. In the current intake set, the most immediate path is the consent-based inbox-to-CRM hygiene service, with the autonomous listing service as the primary redundant track.
